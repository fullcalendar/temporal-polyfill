# WONTFIX for now
# Temporal plain objects can represent dates at the edge of Temporal's ISO
# range, including instants that are outside JavaScript Date's TimeClip range.
# ECMA-402's Temporal bridge formats those plain values from Temporal format
# records / epoch nanoseconds and must not apply Date TimeClip.
#
# This polyfill's Intl.DateTimeFormat wrapper currently does a narrower bridge:
# it converts Temporal.PlainDate, Temporal.PlainDateTime, and
# Temporal.PlainYearMonth to epoch milliseconds and then delegates to native
# Intl.DateTimeFormat format-like methods with those numbers. Native Intl treats
# number inputs like Date-time values, so out-of-TimeClip values throw
# RangeError: Invalid time value before the formatter can produce string output
# or parts.
#
# Fixing these correctly requires a larger Temporal-aware formatting path for
# format, formatToParts, formatRange, and formatRangeToParts. A small patch that
# clamps, offsets, or string-rewrites native Intl output would be wrong because
# the tests cover calendar-sensitive fields, weekdays, ranges, and numbering
# systems. Keep these expected failures grouped until the bridge no longer
# relies on Date-bounded epoch milliseconds for Temporal plain objects.

# WONTFIX for now
# Native named-time-zone transition search currently uses a coarse fixed-size
# sampling period (`periodDur`) before bisecting to an exact transition second.
# The package used a 60-day period before the Stage 4 test262 push, and we are
# intentionally restoring that value for now because it keeps full all-zone
# boundary scans fast.
#
# The known downside is that offset transitions which happen close together can
# be invisible to the current algorithm. If the offset at both ends of a 60-day
# period is the same, the code assumes the whole period is stable and never
# bisects inside it. That misses real tzdb histories where a zone changes away
# from an offset and then changes back within the same period. In the
# representative failing case, Africa/Tunis has two April 1943 transitions eight
# days apart; `getTimeZoneTransition("next")` from before the first one skips to
# the second transition instead of returning the first.
#
# Do not paper over this by shrinking `periodDur` again as the long-term fix.
# The next phase should replace the fixed-window scan with an algorithm that is
# both fast at the representable instant boundaries and able to find multiple
# transitions inside a broader search range. This expected failure is the
# acceptance test for that redesign.
intl402/Temporal/ZonedDateTime/prototype/getTimeZoneTransition/transitions-close-together.js
